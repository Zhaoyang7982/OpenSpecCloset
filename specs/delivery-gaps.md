# 已知交付缺口与验收边界

本文件将逆向阶段识别的 **诚实缺口** 固化为规格边界，供 OpenSpec 对照实验写验收矩阵时使用。

## GAP-A 新手引导（占位）

- **现状**：独立页面存在，内容为占位级。  
- **验收边界**：  
  - **应验收**：入口可达、占位说明不误导用户以为流程已完整。  
  - **不应验收为已完成**：完整引导步骤、可跳过策略、与业务数据联动的引导。

## GAP-B 旅行模式（占位）

- **现状**：独立页面存在，内容为占位级。  
- **验收边界**：  
  - **应验收**：入口可达、占位说明合理。  
  - **不应验收为已完成**：旅行场景业务规则、数据对象、与衣橱/日历的联动。

## GAP-C 用户档案维护闭环

- **现状**：数据结构与推荐消费 **可能已预留**；主界面 **未必** 提供完整档案维护闭环。  
- **验收边界**：  
  - 在矩阵中单列：**档案数据结构 / 推荐侧消费预留** vs **完整维护 UI 与闭环**。  
  - 不得将「字段存在」自动记为「用户可完整维护并生效于推荐」。

## GAP-D 日历备忘与提醒的同步语义

- **现状**：备忘与提醒明确为 **仅存本机**，与云端搭配计划独立。  
- **验收边界**：用户教育文案在多入口 **一致**；换端不带走提醒时 **不** 记为缺陷，除非产品承诺了同步。

## 与「约定业务能力」的关系

- **约定业务能力**：在各 `requirements/*.md` 中以 REQ 条文描述目标行为。  
- **待补齐边界**：以 GAP 与 REQ 中显式「占位/缺口」段落标注；验收时 **区分**「规格已约定」与「当前交付未声称完成」。
